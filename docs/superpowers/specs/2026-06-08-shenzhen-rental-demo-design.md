# 深圳合租点推荐器 Web Demo 设计说明

日期：2026-06-08
状态：面试 demo 设计稿
目标：快速开发一个可演示、可讲清楚推荐逻辑的前端页面

## 1. 背景

面试题要求快速开发一个 Web demo：为想在深圳合租、但在不同地方上班或上学的人，推荐最适合他们的合租点。题目没有限制具体功能和推荐算法，因此 demo 的重点不是做完整租房平台，而是展示产品拆解能力、推荐规则设计能力和前端落地能力。

本 demo 选择“面试展示型”方案：做一个可操作的前端页面，内置深圳样例地点、候选合租区域和模拟通勤数据。页面视觉上展示地图、路线和 API 测算状态，但首期不依赖真实地图 API Key，避免演示现场因为外部服务、鉴权或网络问题失败。

## 2. Demo 一句话定位

为多个在深圳不同地点上班或上学的人，推荐一个在通勤公平、租金适配和生活便利之间最容易达成共识的合租区域。

核心观点：

> 这个工具不是简单找离某个人最近的地方，而是在多人之间做“通勤公平 + 租金适配 + 生活便利”的折中推荐。

## 3. 目标用户

核心用户：

- 2-4 个准备一起在深圳合租的人。
- 每个人有不同上班或上学地点。
- 大家希望找到一个对所有人都相对公平、租金可接受、生活便利的合租区域。

典型场景：

- 两个朋友分别在南山科技园和福田 CBD 上班。
- 三个同学/同事一个在深圳大学，一个在腾讯滨海大厦，一个在会展中心附近。
- 用户不想逐个查地图，而是希望系统给出几个可比较的区域建议。

## 4. 产品范围

### 4.1 首期包含

- 成员输入与编辑。
- 深圳常见上班/上学地点选择。
- 候选合租区域展示。
- 模拟地图视图。
- 模拟路线和通勤时间计算。
- 推荐区域排序。
- 每个推荐结果的解释。
- 权重调节。
- 演示模式与“地图 API 接入能力”展示。

### 4.2 首期不包含

- 真实房源抓取。
- 真实租房交易。
- 用户登录。
- 聊天或室友匹配。
- 支付。
- 后台管理系统。
- 强依赖真实地图 API Key。

## 5. 页面结构

页面采用工作台布局：

左侧为输入与偏好，右侧为地图和推荐结果。

### 5.1 左侧输入区

模块：

- 合租成员列表。
- 添加成员按钮。
- 每个成员可设置：
  - 姓名。
  - 上班/上学地点。
  - 最大可接受通勤时间。
  - 个人优先级。
- 推荐偏好权重：
  - 通勤公平。
  - 平均通勤。
  - 租金适配。
  - 生活便利。
  - 地铁可达。
- 计算按钮。

### 5.2 右侧地图区

模块：

- 深圳简化地图画布。
- 成员目的地 marker。
- 候选合租区域 marker。
- 当前推荐区域高亮。
- 从推荐区域到各成员目的地的路线线条。
- API 状态标签：
  - Demo API mode。
  - Simulated AMap commute matrix。

说明：

地图首期使用前端 SVG / CSS 绘制深圳示意图，不接真实地图瓦片。这样可以稳定展示地图能力，同时保持开发速度。

### 5.3 推荐结果区

模块：

- Top recommendation 卡片。
- 综合分。
- 推荐理由。
- 每个人通勤时间。
- 平均通勤、最差通勤、公平性指数。
- 候选区域对比表。
- 算法解释面板。

## 6. 模拟地图 API 设计

为了让 demo 看起来具备真实 API 接入能力，但不依赖真实 Key，前端实现一个 `MapAdapter` 抽象。

### 6.1 DemoMapAdapter

首期使用 `DemoMapAdapter`，它从本地数据中读取：

- 地点坐标。
- 候选区域坐标。
- 区域到目的地的模拟通勤时间。
- 模拟距离。
- 推荐路线折线点。

它提供与真实地图服务类似的方法：

- `searchPlace(keyword)`：返回地点候选。
- `getCommute(origin, destination, mode)`：返回通勤时间、距离和路线。
- `getCommuteMatrix(origins, destinations)`：批量计算多个区域到多个目的地的通勤矩阵。

### 6.2 FutureAmapAdapter

页面中保留真实高德地图 API 接入说明，但不在首期调用：

- 接入 JS API Key 和 securityJsCode。
- 使用地点搜索或地理编码获取坐标。
- 使用路线规划能力计算公交/地铁、步行或驾车通勤。
- 将真实 API 返回结果转换成和 `DemoMapAdapter` 一样的数据结构。

面试讲述方式：

> 为了保证 demo 稳定，我先把地图能力抽象成 adapter。现在跑的是 DemoMapAdapter，未来只需要替换成 AMapAdapter，推荐算法和页面展示不需要重写。

## 7. 推荐算法

每个候选合租区域都会生成一个综合分。

### 7.1 输入

- 成员目的地。
- 候选合租区域。
- 每个区域到每个目的地的通勤时间。
- 区域租金等级。
- 区域生活便利度。
- 区域地铁可达性。
- 用户权重设置。

### 7.2 核心指标

平均通勤：

- 所有成员通勤时间的平均值。
- 越短越好。

最差通勤：

- 所有成员中最长的通勤时间。
- 如果超过成员可接受上限，则扣分。

通勤公平：

- 衡量不同成员通勤时间差距。
- 差距越小越公平。
- 用最大通勤时间减最小通勤时间表示差距。

租金适配：

- 根据区域租金等级与用户预算偏好计算。
- demo 中使用内置区域租金等级，不接真实房源价格。

生活便利：

- 内置评分，表示商业、餐饮、交通、生活服务成熟度。

地铁可达：

- 内置评分，表示区域地铁线路和换乘便利程度。

### 7.3 默认评分公式

```text
总分 =
通勤公平分 * 35%
+ 平均通勤分 * 25%
+ 最差通勤惩罚后得分 * 15%
+ 租金适配分 * 10%
+ 生活便利分 * 10%
+ 地铁可达分 * 5%
```

### 7.4 推荐解释规则

系统不只展示分数，还要解释原因。

示例：

- “推荐车公庙 / 香蜜湖，因为它在南山和福田之间折中，三人的通勤差距较小。”
- “科技园平均通勤短，但对福田成员不够公平，因此排名低于车公庙。”
- “宝安中心租金更友好，但最差通勤超过 60 分钟，因此不作为首推。”

## 9. 关键交互

1. 用户选择或编辑 2-4 个成员。
2. 用户为每个成员选择上班/上学地点。
3. 用户调整偏好权重。
4. 点击“计算推荐”。
5. 页面显示 API 模拟测算过程。
6. 地图高亮推荐区域和路线。
7. 推荐列表按总分排序。
8. 点击某个候选区域后，地图和解释面板同步切换。

## 10. 错误和空状态

- 少于 2 个成员时，提示“至少需要 2 个合租成员才能做公平推荐”。
- 成员没有选择目的地时，提示补全地点。
- 权重总和不为 100% 时，前端自动归一化。
- 模拟 API 延迟时显示 loading 状态。
- 没有可接受区域时，展示最接近结果，并说明哪些条件过于严格。

## 11. 技术设计

首期实现为前端单页应用。

建议结构：

- `index.html`：页面入口。
- `src/main.js`：应用状态和交互。
- `src/data/shenzhenData.js`：内置地点、候选区域和模拟通勤矩阵。
- `src/services/mapAdapter.js`：地图能力抽象和 DemoMapAdapter。
- `src/services/recommendationEngine.js`：推荐评分算法。
- `src/styles.css`：页面样式。

如果使用构建工具，推荐 Vite；如果时间非常紧，也可以使用原生 HTML/CSS/JS。

## 12. 验收标准

Demo 可交付时应满足：

- 页面可以本地打开或通过本地 dev server 运行。
- 用户可以修改成员地点和偏好权重。
- 点击计算后会展示模拟通勤测算过程。
- 至少展示 5 个候选合租区域的排序。
- Top recommendation 有明确推荐理由。
- 每个候选区域展示每个成员的通勤时间。
- 地图区域能显示成员点、候选区域点和路线。
- 页面明确展示“Demo API mode / Simulated AMap commute matrix”。
- 代码中有清晰的 `MapAdapter`，未来可替换真实地图 API。
- 移动端和桌面端布局基本可用。

## 13. 面试讲述主线

可以这样讲：

1. 先把问题拆成多人通勤公平问题，而不是单点距离最短问题。
2. 再把地图能力抽象成 adapter，保证 demo 稳定，同时保留真实 API 接入路径。
3. 推荐算法不仅输出分数，还输出解释，帮助合租成员达成共识。
4. 首期不接真实房源和交易，避免 scope 膨胀，优先验证核心推荐价值。

一句话总结：

> 我做的不是一个找房列表，而是一个帮助多人合租做地点决策的推荐工具。
