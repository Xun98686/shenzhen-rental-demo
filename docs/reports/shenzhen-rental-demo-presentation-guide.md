# 深圳合租点推荐器 Web Demo 讲解报告

## 1. 一句话介绍

这个 Web demo 是一个“多人合租地点决策工具”。它帮助 2-4 个在深圳不同地点上班或上学的人，根据通勤公平、平均通勤、最差通勤、租金适配、生活便利和地铁可达性，推荐最适合一起合租的区域。

推荐讲法：

> 我没有把它做成普通找房列表，而是先解决合租前最难的决策问题：大家住在哪里才相对公平、可接受、容易达成共识。

## 2. 为什么做这个方向

老板给的题目是“为想在深圳合租，但在不同地方上班上学的人推荐最适合的合租点”。这个问题的核心不是单纯找最近位置，而是多人之间的折中。

如果只按“平均距离最短”推荐，可能会出现一个人通勤 10 分钟、另一个人通勤 70 分钟的情况。这样的结果虽然数学上看起来不错，但现实里很难让合租成员接受。

所以这个 demo 的核心判断是：

- 合租推荐要考虑公平性，而不只是最短距离。
- 推荐结果要能解释，否则用户不容易信任。
- demo 阶段要保证稳定展示，所以用模拟通勤矩阵代替真实地图 API。

## 3. 页面功能怎么讲

可以按页面从左到右讲。

### 左侧：输入合租成员

左侧是输入区，用户可以设置：

- 合租成员姓名。
- 每个人上班或上学地点。
- 每个人最大可接受通勤时间。
- 推荐偏好权重。

可以这样讲：

> 这里模拟真实用户的决策输入。每个人的目的地和可接受通勤时间不同，所以系统不能只找一个地理中心点，而要计算每个候选区域对每个人是否公平。

### 右侧上方：模拟地图和通勤测算

右侧上方是深圳示意地图，展示：

- 候选合租区域。
- 成员上班/上学地点。
- 推荐区域到每个目的地的路线。
- API ready / 模拟通勤 API 测算中的状态。

可以这样讲：

> 这里展示的是地图 API 能力的产品形态。当前 demo 使用内置深圳数据模拟通勤时间，目的是保证面试现场稳定。真实上线时，可以把这一层替换为高德地图或其他地图服务的路线规划接口。

注意：页面上不需要主动强调 DemoMapAdapter 这个技术名词。如果面试官问技术实现，再解释“我把地图能力做成 adapter 抽象”。

### 右侧中部：推荐结果

系统会给出 Top recommendation，比如“车公庙 / 香蜜湖”，并展示：

- 综合评分。
- 推荐理由。
- 平均通勤。
- 最差通勤。
- 通勤差距。
- 租金等级。

可以这样讲：

> 推荐结果不只是一个分数，还会解释为什么推荐。比如车公庙 / 香蜜湖不是对某个人最近，但它在南山和福田之间比较均衡，所以更容易被合租成员接受。

### 右侧下方：候选区域对比

候选区域表格展示多个区域的分数、平均通勤、最差通勤和租金等级。点击不同区域，地图和推荐说明会同步切换。

可以这样讲：

> 我保留候选区域对比，是因为租房决策通常不是系统说一个答案就结束。用户需要看到备选方案，比较为什么第一名更合适、第二名差在哪里。

## 4. 推荐算法怎么讲

推荐算法可以用简单公式讲，不要讲得太学术。

默认总分：

```text
总分 =
通勤公平分 * 35%
+ 平均通勤分 * 25%
+ 最差通勤分 * 15%
+ 租金适配分 * 10%
+ 生活便利分 * 10%
+ 地铁可达分 * 5%
```

每个指标的含义：

- 通勤公平分：看每个人通勤时间差距，差距越小越好。
- 平均通勤分：看整体通勤效率，平均时间越短越好。
- 最差通勤分：防止某一个成员被牺牲，最长通勤过高会扣分。
- 租金适配分：用区域租金等级模拟预算适配。
- 生活便利分：模拟餐饮、商业、日常服务成熟度。
- 地铁可达分：模拟地铁线路和换乘便利程度。

推荐讲法：

> 我把问题拆成“效率”和“公平”两个维度。平均通勤代表效率，通勤差距和最差通勤代表公平。合租场景里公平很重要，所以我给通勤公平更高权重。

## 5. 为什么没有接真实地图 API

这个问题面试官很可能会问，建议这样回答：

> 因为这是快速 demo，我优先保证功能闭环和现场稳定。真实地图 API 会涉及 Key、配额、网络、鉴权和跨域等不确定因素，容易让面试展示翻车。所以我先用内置深圳通勤矩阵模拟 API 返回结果，同时把地图能力单独抽象出来，未来接高德地图时只需要替换数据来源。

再补一句：

> 这个 demo 的重点不是证明我会调 API，而是证明我能把业务问题拆成产品流程、推荐规则和可演示的前端体验。

## 6. 演示流程建议

面试时可以按 5 步演示：

1. 先介绍场景：几个人想在深圳合租，但目的地分散。
2. 修改左侧成员地点，比如一个在南山，一个在福田，一个在深圳大学。
3. 点击“计算推荐”，展示模拟 API 测算状态。
4. 看首推区域，解释为什么不是简单选离某个人最近的地方。
5. 点击候选区域表格，展示不同区域的取舍。

演示时可以说：

> 这里我点击不同候选区域，右侧地图和通勤数据会一起变化。这样用户不是被动接受推荐，而是能理解每个区域的 trade-off。

## 7. 可以主动强调的产品亮点

- 不是房源列表，而是合租前的地点决策工具。
- 不只算平均距离，还考虑成员之间的公平性。
- 每个推荐都有解释，增强用户信任。
- 页面展示了地图 API 接入形态，但 demo 不依赖真实 API。
- 通过权重滑块，用户可以表达自己更重视通勤、租金还是生活便利。
- 留出了未来扩展空间：真实地图 API、真实房源、预算、室友偏好、地铁线路等。

## 8. 面试官可能追问的问题

### Q1：为什么首推车公庙 / 香蜜湖？

可以回答：

> 因为在默认样例里，成员目的地分布在南山、深圳大学和福田 CBD。车公庙 / 香蜜湖虽然不是某一个人的最近点，但它让三个人的通勤差距相对小，最差通勤也可控，所以综合分最高。

### Q2：如果真实上线，你会怎么接地图 API？

可以回答：

> 我会把现在模拟的通勤矩阵替换成真实地图服务。具体步骤是：地点搜索或地理编码拿坐标，路线规划 API 算每个候选区域到每个目的地的通勤时间，再把结果喂给现有推荐算法。这样 UI 和推荐引擎不用重写。

### Q3：为什么不直接推荐两点中间？

可以回答：

> 两点中间不一定交通方便，也不一定租金合适。合租地点要考虑地铁换乘、区域生活便利、租金和每个人的最大可接受通勤，所以不能只看几何中心。

### Q4：这个 demo 下一步怎么扩展？

可以回答：

> 下一步可以接真实地图 API 和真实房源数据，再加入预算、房型、通勤方式偏好、地铁线路偏好和室友偏好。第一阶段我先验证核心价值：帮助多人快速达成合租区域共识。

## 9. 30 秒版本讲稿

如果时间很短，可以这样讲：

> 这个 demo 解决的是多人合租前的选址问题。用户输入每个人的上班或上学地点、最大可接受通勤时间和偏好权重，系统会基于模拟地图通勤矩阵，对深圳多个候选区域打分。评分不只看平均通勤，还重点考虑通勤公平和最差通勤，避免某一个成员被牺牲。页面会给出推荐区域、每个人预计通勤、候选区域对比和推荐解释。当前为了演示稳定使用模拟 API 数据，未来可以替换成真实高德地图路线规划接口。

## 10. 2 分钟版本讲稿

如果有完整讲解时间，可以这样讲：

> 我先把题目拆成一个多人决策问题。合租选址不是找离某个人最近的地方，而是让所有成员在通勤、租金和生活便利之间达成共识。  
>
> 页面左侧是输入区，可以设置合租成员、上班上学地点、最大通勤时间和偏好权重。右侧是结果区，包含深圳示意地图、通勤路线、首推区域、成员通勤时间和候选区域对比。  
>
> 推荐算法由六部分组成：通勤公平、平均通勤、最差通勤、租金适配、生活便利和地铁可达。其中通勤公平权重最高，因为合租场景里最怕一个人特别方便、另一个人被迫通勤很远。  
>
> 当前 demo 使用模拟地图通勤矩阵，是为了保证面试现场稳定。但代码结构上把地图能力和推荐算法分开，未来接真实地图 API 时，只要把通勤时间数据来源替换掉，推荐逻辑和页面展示可以继续复用。  
>
> 所以这个 demo 展示的不是一个简单房源列表，而是一个帮助多人做合租地点决策的推荐工具。
